# KG 模块：命名规范与质量审查

> **负责人**：成员 I
> **说明**：本文档制定全组统一的知识图谱命名规范，并在各成员提交 KG 文档后进行一致性审查。

---

## 一、命名规范

### 1.1 知识点命名

- **使用中文名**，≤10 字
- **避免歧义**：如"堆"明确为"堆（数据结构）"
- **统一简称**：BST→二叉搜索树，AVL→平衡二叉树
- **首字母大写**否（中文即可）

### 1.2 分类命名（category）

统一使用以下英文分类名：

| 分类 | 值 | 说明 |
|------|-----|------|
| 线性结构 | `linear` | 顺序表、链表、栈、队列 |
| 树形结构 | `tree` | 二叉树、BST、AVL、堆、哈夫曼树 |
| 图形结构 | `graph` | 图的存储、遍历、MST、最短路径、拓扑排序 |
| 排序算法 | `sort` | 冒泡、快速、归并、堆排序 |
| 查找算法 | `search` | 顺序查找、二分查找、哈希表 |
| 基础概念 | `other` | 算法复杂度、递归 |

### 1.3 节点 ID 格式

```
kp_{三位序号}
示例：kp_001, kp_002, ..., kp_030
```

ID 分配方案（由成员 I 在汇总时统一分配）：

| ID 范围 | KG 模块 | 负责成员 |
|--------|---------|---------|
| kp_001 ~ kp_007 | 线性结构 | A |
| kp_008 ~ kp_010 | 树形结构 Part1 | B |
| kp_011 ~ kp_013 | 树形结构 Part2 | C |
| kp_014 ~ kp_016 | 图形结构 Part1 | D |
| kp_017 ~ kp_019 | 图形结构 Part2 | E |
| kp_020 ~ kp_024 | 排序算法 | F |
| kp_025 ~ kp_028 | 查找算法 | G |
| kp_029 ~ kp_030 | 基础概念 | H |

### 1.4 难度评级标准（全体统一）

| 难度 | 标准 | 示例 |
|------|------|------|
| 1 | 纯概念介绍，无需编程基础 | 算法复杂度、冒泡排序、顺序查找 |
| 2 | 简单结构/基本操作，单一数据结构 | 顺序表、链表、栈、队列、二叉树 |
| 3 | 需要组合多个概念，算法有变体 | 二叉搜索树、图的遍历、快速排序、归并排序 |
| 4 | 算法较复杂，需要数学基础 | AVL树、最短路径、最小生成树、哈希表 |
| 5 | 高级主题（本期暂不涉及） | B+树、红黑树、NP完全问题 |

### 1.5 关系类型规范

| 关系类型 | Cypher 标签 | 方向 | 含义 |
|---------|------------|------|------|
| 前置依赖 | `PREREQUISITE_OF` | 前置 → 后继 | 学习 B 之前需先掌握 A |
| 层级包含 | `SUBTOPIC_OF` | 子 → 父 | 子知识点是父知识点的细分 |
| 跨类应用 | `APPLIED_IN` | 工具 → 场景 | 某结构/算法应用于某场景 |
| 相关关系 | `RELATED_TO` | 双向 | 两个知识点有概念关联 |

## 二、质量审查清单

在各成员提交 KG 文档后，按以下清单逐项审查：

- [x] 所有知识点名称无重复、无歧义
- [x] 所有知识点已分配难度（1~5）和分类（英文）
- [x] 每个 PREREQUISITE_OF 关系是否合理（B 是否真的依赖 A？）
- [x] 每个 SUBTOPIC_OF 关系是否准确（子节点是否确实是父节点的细分？）
- [x] 跨模块关系是否正确对接（A 模块声明的依赖在 B 模块中是否存在对应节点？）
- [x] 是否存在遗漏的重要知识点（如遗漏了"循环链表"？）
- [x] 命名是否符合统一规范
- [x] 难度评级是否与其他模块一致（同一难度等级的知识点难度是否相当？）
